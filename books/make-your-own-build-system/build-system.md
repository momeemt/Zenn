title: "ビルドシステムとは"
---

`cargo build`、`npm run build`、`make` など、言語やプロジェクトによって様々なビルドコマンドが存在します。

これらのコマンドの裏側で動いているのがビルドシステムです。ビルドシステムとは、何らかの入力から出力（成果物）を得るための工程を自動化するツールです。

たとえば、C言語のソースコードから実行可能ファイルを生成する場合を考えます。

```
hello.c → [コンパイル] → hello.o → [リンク] → hello
```

1つのファイルだけならコマンドを手で打てば済みますが、実際のプロジェクトでは数百、数千のファイルを扱います。どのファイルをどの順番で、どのようにコンパイルするか。これを人間が毎回考えるのは現実的ではありません。

ビルドシステムは、この複雑な工程を定義ファイル（Makefile、build.gradle、Cargo.toml など）に基づいて自動実行します。

## なぜビルドシステムが必要なのか

ビルドシステムがない世界を想像してみてください。10個のCファイルからなるプロジェクトをビルドするには、次のようなコマンドを毎回入力する必要があります。

```bash
gcc -c -o main.o main.c
gcc -c -o utils.o utils.c
gcc -c -o network.o network.c
gcc -c -o parser.o parser.c
gcc -c -o config.o config.c
gcc -c -o logger.o logger.c
gcc -c -o auth.o auth.c
gcc -c -o db.o db.c
gcc -c -o cache.o cache.c
gcc -c -o app.o app.c
gcc -o myapp main.o utils.o network.o parser.o config.o logger.o auth.o db.o cache.o app.o
```

これには以下のような問題があります。

1. 毎回同じコマンドを打つのは面倒で、ミスも起きやすい
2. 1つのファイルだけ変更しても、全ファイルを再コンパイルしてしまう
3. コンパイルの順序を間違えるとエラーになる
4. 新しいファイルを追加するたびにコマンドを修正する必要がある

ビルドシステムは、これらの問題を解決します。定義ファイルに何をどうビルドするかを記述しておけば、`make` や `ninja` といったコマンド一発でビルドが完了します。

## 代表的なビルドシステム

いくつかの代表的なビルドシステムを紹介します。それぞれ設計思想や得意分野が異なるため、用途に応じて使い分けられています。

### Make（1976年〜）

Stuart Feldman が Bell Labs で開発した、最も歴史のあるビルドシステムの1つです[^make-history]。`Makefile` にターゲット、依存関係、ビルドコマンドを記述します。

[^make-history]: Make は UNIX の歴史とともに歩んできました。現在最も広く使われている実装は GNU Make です。

```makefile
hello: hello.o
	gcc -o hello hello.o

hello.o: hello.c
	gcc -c hello.c
```

シンプルですが、大規模プロジェクトでは記述が煩雑になりがちです。また、Makefile の文法には独特の癖があり、タブとスペースの違いでエラーになるなど、初心者には難しい面もあります[^makefile-tab]。

[^makefile-tab]: Makefile ではインデントにタブ文字を使う必要があります。スペースを使うとエラーになります。これは Make が開発された1970年代の設計判断がそのまま残っているためです。

### CMake（2000年〜）

Makefile を直接書く代わりに、より抽象的な `CMakeLists.txt` を書き、そこから Makefile や Ninja のビルドファイルを生成します。このようなツールをビルドシステムジェネレータと呼びます。

```cmake
cmake_minimum_required(VERSION 3.10)
project(hello)
add_executable(hello hello.c)
```

クロスプラットフォーム対応が強みで、Windows、macOS、Linux で同じ `CMakeLists.txt` からそれぞれの環境に適したビルドファイルを生成できます。C/C++ プロジェクトでは事実上の標準となっています。

### Ninja（2012年〜）

Google Chrome の開発チームが、ビルド速度を追求して開発したビルドシステムです[^ninja]。人間が直接記述することは想定しておらず、CMake などのビルドシステムジェネレータから出力されることを前提としています。

[^ninja]: Ninja は「最速のビルドシステム」を目指して設計されました。Make と比較して、特に大規模プロジェクトでのインクリメンタルビルドが高速です。

### Bazel（2015年〜）

Google が開発した、大規模プロジェクト向けのビルドシステムです。再現性のあるビルドと分散ビルドを重視しており、モノレポ（巨大な単一リポジトリ）での運用に適しています。

Google 社内では、数十億行のコードを含む単一リポジトリを Bazel（の社内版である Blaze）でビルドしています[^google-monorepo]。

[^google-monorepo]: Google のモノレポについては「Why Google Stores Billions of Lines of Code in a Single Repository」という論文で詳しく解説されています。

### 言語固有のビルドシステム

多くのプログラミング言語は、独自のビルドシステムを持っています。

| 言語 | ビルドシステム/パッケージマネージャ |
| --- | --- |
| Rust | Cargo |
| Go | go build |
| Java | Gradle, Maven |
| JavaScript | npm, yarn, pnpm |
| Python | pip, poetry |

これらは単なるビルドだけでなく、依存関係の管理やパッケージの公開機能も備えています。

## ビルドシステムの高度な機能

現代のビルドシステムには、単にコマンドを実行するだけでなく、様々な高度な機能が備わっています。この講義では、これらの機能を実際に実装していきます。

### インクリメンタルビルド

前回のビルド以降に変更されたファイルや、その影響を受ける部分だけを再ビルドする機能です。

たとえば、100個のソースファイルがあるプロジェクトで1つだけ変更した場合、全体を再ビルドする必要はありません。変更されたファイルと、それに依存するファイルだけをビルドすれば十分です。

大規模プロジェクトでは、この機能がないとビルドのたびに数十分待つことになります。開発効率に直結する重要な機能です。

### 依存関係の解決

複数のファイルやライブラリが互いに依存している場合、正しい順序でビルドする必要があります。ビルドシステムは依存関係を解析し、適切な順序を自動的に決定します。

これは数学的にはグラフ理論の問題で、トポロジカルソートというアルゴリズムを使います。第4章で詳しく扱います。

### ロックファイル

依存するライブラリのバージョン、取得元、チェックサムなどを記録したファイルです。npm の `package-lock.json` や Cargo の `Cargo.lock` がこれにあたります。

ロックファイルがあることで、異なる環境・異なるタイミングでも同じバージョンの依存関係を再現できます。「昨日は動いていたのに今日は動かない」という問題を防ぎます[^works-on-my-machine]。

[^works-on-my-machine]: 「自分のマシンでは動く（Works on my machine）」は、エンジニア間でよく使われるジョークです。ロックファイルはこの問題を解決する重要なツールです。

### ビルドキャッシュ

以前のビルドで生成した成果物を保存しておき、同じ入力からは再計算せずに成果物を取得する仕組みです。CI/CD パイプラインで特に効果を発揮します。

たとえば、GitHub Actions で毎回フルビルドするのではなく、変更のない部分はキャッシュから取得することで、CI の実行時間を大幅に短縮できます。

### 再現性のあるビルド

同じ入力・同じ環境条件でビルドした場合、必ず同じ出力を得られるビルドのことです。Nix はサンドボックスを使ってビルド環境を隔離し、この再現性を追求しています。

再現性のあるビルドは、セキュリティの観点からも重要です。ソースコードからバイナリが正しく生成されたことを第三者が検証できるからです[^reproducible-builds]。

[^reproducible-builds]: Debian プロジェクトでは、パッケージの再現可能ビルドを推進しています。詳しくは https://reproducible-builds.org/ を参照してください。

## 本書で作るもの

この本では、minimake という小さなビルドシステムを作ります。

最初はシンプルに JSON で定義されたコマンドを順番に実行するだけのものから始め、段階的に機能を追加していきます。

- 第3章: 単純なビルドの実行
- 第4章: 依存関係の解決（トポロジカルソート）
- 第5章: インクリメンタルビルド

Python の標準ライブラリのみで実装します[^python-stdlib]。

[^python-stdlib]: 外部ライブラリを使わないことで、環境構築の手間を省き、ビルドシステムの本質的な部分に集中できます。

:::message
この講義では、ビルドシステムの概念を理解することを重視しています。実装するコードは最小限に抑え、各機能の本質を理解することを目指します。
:::

それでは、実際にコードを書いていきましょう。
